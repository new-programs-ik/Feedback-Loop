1.  What this module does
This is the first piece we are building. Its job is simple: every Monday, take the ratings from Sunday’s classes, find the classes that did not go well, and get clear, helpful feedback to those instructors. The system does the boring, repeated work — collecting the ratings, sorting them, and writing a first draft of the feedback. The PM stays in charge of the parts that need a human: approving the words and choosing to send. The aim is to save the PM from spending hours checking every class by hand.
2.  When it runs
Most classes happen on Sunday, and that is the main day. Some classes also run on other days (for example Thursday or Saturday), so the module looks at the whole week’s classes — but only for the courses you are responsible for. It does not scan every class in Metabase, just yours. The ratings are ready by Monday morning, around 9 AM IST. So the module runs first thing Monday, and by the time the PM sits down, the report and short list are already waiting.

5.  Step by step, in detail
Step 1 — Collect the ratings  (automatic)
On Monday morning (around 9 AM IST), the system reads the new ratings from Metabase (the tool where ratings are stored), for the courses you are responsible for — Flagship ML, Advanced ML, PwC Accelerator and FDE. It does this on its own, on a timer. No one has to start it.
Step 2 — Sort and judge  (automatic)
The system groups the ratings by course, then by class, then by instructor. Each course is judged on its own — Flexible classes are compared with Flexible, Advanced ML with Advanced ML, and the same for PwC and FDE. Then it applies one simple rule: 4.5 out of 5 or higher is good; below 4.5 needs attention.
Step 3 — The short list  (automatic)
The system shows the PM a short list of only the low-rated classes — the ones below 4.5, lowest first. The PM no longer scrolls through every class in every cohort. They see just the ones that need a look.
Step 4 — Get the transcript  (a person does this, for now)
For each low-rated class, the system needs the class transcript so the AI can see what happened. Our recordings live on Vimeo (vimeo.com). For now the PM gets the transcript by hand: log in to Vimeo with the IK email, open the class using the class link, go to Manage → Subtitles, open the three-dot menu, and download the transcript — Vimeo offers .vtt, .srt and .ttml formats. The PM then uploads that file into the card (shown in Section 7). With only two or three low-rated classes in a normal week, this is quick. Later it becomes automatic (see Section 10), because Vimeo has an API that can pull a video’s captions directly.
Step 5 — The AI drafts the feedback  (automatic)
The AI reads the transcript and writes a first draft of the feedback. It is specific and kind: what went well, what could be better, and concrete suggestions (for example, “slow down on the tree-based part and pause for questions”). This is the one place we use AI — to read and write a draft, not to decide anything.
Step 6 — The PM checks and approves  (always a person)
The draft goes to the PM. They read it, change anything they want, and approve it. Nothing is ever sent without this step. Feedback about a real person’s teaching is sensitive, so a human always has the final say on the words and on whether to send.
Step 7 — Send it and save a record  (automatic)
Once the PM approves, the system sends the feedback to the instructor on Discord. It also saves a full record: which class, the rating, the transcript used, the draft, the PM’s edits, the approval, and the time it was sent. Nothing is hidden, and anyone can look back later.
6.  The Monday report — one Excel file, four tabs
This file is the shared store for ratings — it is where the data pulled from Metabase is kept, so the automation and the PMs work from the same place. Every Monday the system writes that week’s classes for your courses into it (all of them, not only the low ones), flags the low-rated ones (below 4.5), and updates each row’s status as it works (transcript needed → draft ready → sent). It is one Excel file with four tabs, one per course: Flagship ML, Advanced ML, PwC Accelerator and FDE. Importantly, every row carries the date of the class, so it is always clear which day’s rating it is. The columns are: date, cohort, instructor, class, rating, number of ratings, status, and the Vimeo class link (so the transcript is one click away). Two counts at the top show how many classes there were and how many were low-rated.
A sample of this file is attached so you can see the exact shape. Each row is one class. The low-rated ones are flagged in red — for example, Justin’s “Classification Techniques” class for the Early August 2026 Flagship cohort, rated 4.47, which is below the line — while well-rated classes simply sit in the store as a record.

The app you build — the dashboard and its backend — is the brain. The backend reads the ratings, applies the 4.5 rule, builds the short list, holds the card, waits for the PM’s approval, asks the AI to draft, and writes the record. n8n is the hands — small workflows that reach out to other tools: poll Metabase for ratings, send the message on Discord, alert the PM if needed, and (later) fetch the transcript.

The one place the LLM is used
The LLM is used in exactly one step — drafting the feedback (Step 5). Nowhere else. It reads a transcript and writes a draft; it never decides what to do and never sends anything.
Its task: read one class’s transcript and write a short, useful feedback draft for that instructor.
What it is given (input): the transcript text, plus the class context — course, topic, instructor, the rating and how many ratings — and a simple rubric of what good feedback looks like.
What it returns (output): a short draft in three parts — what went well, what to improve, and two or three concrete suggestions — in a kind, specific, constructive tone. It can also add a flag if the class looks bad enough to consider re-running (a recall), but that is only a suggestion.
Guardrails: it uses only what is actually in the transcript (it does not make things up), it stays practical and respectful, and the PM always edits and approves before anything is sent.
Other technical points
Metabase: read through its API using a saved question; only your courses are pulled, on the Monday schedule.
The sheet is the shared store: the automation writes the pulled ratings and updates each row’s status; the PMs read the same sheet. Later this moves to a database.
Discord: the approved feedback is sent by a Discord bot to the instructor’s channel.
Vimeo (later): the transcript is fetched automatically through Vimeo’s captions API using an IK token.
Audit log: every action is recorded with its reason, so nothing is hidden.

11.  Tricky cases and simple rules
Exactly 4.5: counts as good. Only below 4.5 is flagged.
Few ratings: if a class has only a handful of ratings, the system still flags it but marks it “few ratings”, so the PM knows to weigh it carefully.
Feedback is per class, not per instructor: each low-rated class gets its own feedback, sent to the instructor who took it. If the same instructor taught another class that scored well (4.5 or above), that one is left alone. So if an instructor has two low-rated classes, that is two separate feedback messages.
An instructor who teaches in more than one course: they are judged separately inside each course.
Repeated low ratings in a cohort: this feeds the staffing rule we agreed earlier — a poorly-rated instructor is not put back on the same cohort. (That part lives in the staffing module.)
12.  What we record
For every class this module touches, the system saves: which class and cohort, the instructor, the rating, the transcript used, the AI’s draft, the PM’s edits, the approval, and when and where the feedback was sent. This is the record you can always go back to — nothing happens invisibly.
